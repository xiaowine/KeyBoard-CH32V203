# HID 双向可靠传输协议规范 (V1.2)

## 1. 协议概述
本协议专为固定 32 字节报文长度的 HID 设备设计，旨在提供高可靠性的双向流传输。
- **核心目标**：分包传输、丢包重发、硬件校验、非阻塞执行。
- **硬件优化**：采用 **4 字节手动对齐** 布局，确保 32 位 MCU 访问 `Payload` 和 `CRC32` 时无需非对齐处理，支持 DMA 和硬件 CRC 加速。

---

## 2. 帧结构定义 (Frame Structure)
整帧固定为 **32 字节**。通过 `Reserved` 字节强制对齐。

| 偏移 (Byte) | 字段名 | 长度 | 类型 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 0 | **Ctrl_Len** | 1 | uint8_t | 高 3 位：包类型；低 5 位：有效长度 (0-24) |
| 1 | **DataType** | 1 | uint8_t | 业务负载类型（由应用层定义） |
| 2 | **Seq_Num** | 1 | uint8_t | 序列号 (0-255 循环) |
| 3 | **Reserved** | 1 | uint8_t | **保留字节**：用于 4 字节对齐补位，默认 0x00 |
| 4-27 | **Payload** | 24 | uint8_t[24] | **4 字节对齐起始** 的实际载荷数据 |
| 28-31 | **CRC32** | 4 | uint32_t | 硬件 CRC32 校验值 (覆盖 Offset 0-27) |

---

## 3. 字段详解

### 3.1 Ctrl_Len (控制与长度)
- **Packet Type (Bit 7:5)**:
  - `000` (SINGLE): 单包任务 (数据量 $\le$ 24 字节)。
  - `001` (START): 长数据起始包。
  - `010` (DATA): 长数据中间包。
  - `011` (END): 长数据结束包。
  - `100` (ACK): 确认包 (用于状态机推进)。
  - `101` (NACK): 错误通知 (请求重发)。
  - `110` (BUSY): 设备忙。
  - `111` (OF): 溢出/错误，表示发送端/接收端缓冲区溢出或其他错误，需丢弃当前传输任务。
- **Data Length (Bit 4:0)**: 
  - Payload 字段中有效字节数 ($0 \sim 24$)。

### 3.2 Seq_Num (序列号)
- 物理传输能力：$256 \text{ pkts} \times 24 \text{ bytes} = 6,144 \text{ Bytes}$ (6 KB)。
- 若超过 6KB，应用层需处理序列号回绕 (Rollover)。

---

## 4. 传输逻辑 (10ms 非阻塞状态机)

### 4.1 发送方 (TX)
1. **PREPARE**: 组包并计算硬件 CRC32，通过 HID 发出。
2. **WAIT_ACK**: 启动计时器。若收到匹配 `Seq_Num` 的 ACK 则进入下一包；若超时（建议 50ms）则递增 `retry_cnt` 并重传。
3. **IDLE**: 任务完成或彻底失败。

### 4.2 接收方 (RX)
1. **校验**: 硬件 CRC32 校验失败则丢弃，不予回复。
2. **去重**: 若收到的 `Seq_Num` 等于 `last_seq`，说明 ACK 丢失，**必须重发 ACK** 但丢弃 Payload 数据。
3. **超时**: 若 `START` 后超过 500ms 未收到后续，强制复位至 IDLE。

---

### 4.3 实现细节与约定（来自 C 实现）

- 报文固定长度为 `HID_COMM_DATA_SIZE`（32 字节），其中 Payload 最大为 24 字节，CRC32 占用偏移 28-31（覆盖偏移 0-27）。文档中称 CRC 覆盖范围为 0~27，与实现一致。
- 帧结构在代码中以按位域布局实现：低 5 位为 `len`（0~24），高 3 位为 `type`（帧类型）。接下来的字段为 `data_type`, `seq`, `reserved`, `payload[24]`, `crc32`。
- `START` 包的 payload 前 4 字节为 `total_size`，采用小端序（uint32_t, little-endian）。接收方收到 `START` 后，会把 `total_size` 写入 `total_len` 并进入 `RECEIVE_WAIT` 状态。
- 接收缓冲区：实现中默认使用一个 24 * 256 = 6144 字节 的重组缓冲区，能支持最多 256 帧的连续分包（共 6144 字节）。如果 `total_size` 超过该容量或为 0，则接收方会 NACK 并进入错误态。
- 重试与限制：实现中定义 `HID_RETRY_LIMIT` 为 5（连续重试次数上限）。当接收侧对 seq/类型/长度等出现可重试错误时，会累计 `retry_cnt`，超过上限后丢弃当前接收任务并进入错误态。
- 严格顺序要求：`DATA` / `END` 包要求严格顺序到达，期望 `seq == last_seq + 1`。若不匹配，接收方会发送 `NACK`，并在可重试范围内等待对端重发。
- `END` 包要求累积接收长度正好等于 `total_len`；否则回 `NACK` 并触发重试计数递增。
- ACK/NACK 语义：收到 `ACK` 会使发送端推进到下一帧；收到 `NACK` 则触发发送端的重发逻辑并增加重试计数。实现里还处理 `BUSY` 与 `OF` 控制帧：当远端发送 `BUSY` 或 `OF` 并且 seq 匹配当前发送任务时，发送端会中止当前发送任务（丢弃整个消息）。
- **Seq 限制（新增协议约束）**：
  - 所有通知/传输任务必须以 `seq == 0` 开头。即：
    - 对于单包任务（`SINGLE`），该帧的 `seq` 必须为 0。若不为 0，接收端会返回 `NACK` 并丢弃该帧。
    - 对于多包任务，`START` 帧的 `seq` 必须为 0；后续 `DATA` / `END` 的 `seq` 应依次为 1、2 …（按字节包序递增，模 256 回绕）。
  - 该限制用于简化接收端去重与重传逻辑，避免跨任务序号混淆。
- 接收端对已完成数据并不会自动清空缓冲区：完成态会把数据保留在接收缓冲区中，等待上层应用读取并处理。
- 校验实现：代码使用硬件 CRC 外设计算 7 个 32-bit 字（28 字节）的 CRC（即覆盖偏移 0-27），先调用 `CRC_ResetDR()` 再 `CRC_CalcBlockCRC()`，与协议定义一致。

### 4.4 实际行为要点（快速列表）

- **START payload 前 4 字节 = total_size（小端）**。
- **Payload 单包最大 24 字节，整帧 32 字节**（含 CRC）。
- **接收缓冲上限 6144 字节（24*256）**，超限会 NACK。
- **连续 NACK 超过 5 次（HID_RETRY_LIMIT）后放弃当前接收任务**。
- **收到重复 seq（与 last_seq 相同）时重发 ACK，不重复写入 payload**。
- **收到 BUSY/OF 控制帧时，发送端中止当前发送任务**。

## 5. C 语言实现

### 5.1 协议帧结构
```c
#include <stdint.h>

#pragma pack(push, 1)
/**
 * @brief HID 传输协议帧 (32-byte)
 * 手动对齐以优化 32 位 MCU 性能
 */
typedef struct {
    uint8_t ctrl_len;    // Bit 7-5: Type, Bit 4-0: Len
    uint8_t data_type;   // 业务类型
    uint8_t seq;         // 序列号
    uint8_t reserved;    // 保留补位 (确保后续4字节对齐)
    uint8_t payload[24]; // 负载起始偏移 = 4
    uint32_t crc32;      // CRC32起始偏移 = 28
} HidFrame_t;
#pragma pack(pop)
